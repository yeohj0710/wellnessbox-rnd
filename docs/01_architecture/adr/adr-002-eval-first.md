# ADR-002: eval-first를 기본 개발 순서로 채택한다

- 상태: Accepted
- 날짜: 2026-03-08
- 기준 문서: `C:/dev/wellnessbox-rnd/docs/context/master_context.md`

## 문맥

- 원문 p.25~26 KPI가 실제 성공 기준이다.
- 추천 정확도, 상담 정확도, 안전 근거 정확도, 상태기계 정확도는 구현 전에 측정 구조가 있어야 한다.
- 웹 기능부터 만들면 "보이는 데모"는 빠르게 나오지만 KPI 달성 여부를 모른 채 작업하게 된다.

## 결정

- 모든 주요 엔진 구현 전에 frozen eval 포맷과 리포트 포맷을 만든다.
- 새 모듈은 최소한의 regression case 없이는 배포 후보로 올리지 않는다.
- 배포 게이트는 `성능 체감`이 아니라 `eval 통과 여부`를 따른다.

## 이유

1. KPI는 UI가 아니라 평가 수치로 판정된다.
2. 단일 개발자는 감에 의존한 반복보다 자동 평가가 훨씬 효율적이다.
3. 구조화 규칙 엔진과 상태기계는 eval-first와 궁합이 좋다.

## 결과

- 장점: 회귀 방지, 우선순위 명확화, KPI 중심 개발
- 비용: 초기 문서/테스트 구축 시간이 추가됨
- 후속 조치: `apps/eval_runner`, `data/frozen_eval`, KPI 리포트 규격 정의

## 기각한 대안

- 기능 구현 후 나중에 평가: 이미 잘못된 구조 위에 기능이 누적된다.
- 수동 점검 중심: 재현성이 없다.
